Zenta StyledHtml module

Abstract

  • You have designed something, you have the Zenta model on it. But you have to create a documentation in the form requested by your client.
  • You are designing something complex. You want to make sure that you have thought about all aspects and interdependencies.
  • You work with junior architects or developers. You are tired of checking their models for trivial errors and omissions.
  • You are doing detailed design, and it is boring to do the trivial task of converting the model to config file by hand.

If any of the above happens, this module and the underlying concepts are developed for you.

This module makes possible to

  • define and document enterprise architecture policies
  • check architecture models against those policies
  • define documentation styles for various stakeholders, and generate those documentations from the model
  • generate configuration and code directly from the model

This model tries to fulfill multiple purposes:

- Show how Zenta can be used in the goovernance of IT

- Document the styledhtml Zenta module

- Serve as an example for both of the above

- The webpage of the styledhtml module is generated from it. See http://archistyledhtml.sourceforge.net/

You can find user documentation in the Use Cases section.

1. Introduction

This module is mainly about two things.

One is to create reports from your Zenta model for the different stakeholders. If you are interested in this part, just jump to use cases, where you can find the user documentation.

The other thing is to implement Zenta Extensions. You can find some introductory text here, and an example implementation in the "Sample policy" section.

To see some of the possible avenues it would open, see "Possible enhancements"

1. Quick howto

This introduction is for the impatient, who just wants to generate documentation from an already existing Zenta model.

  • Download the Zenta Styledhtml distribution from http://sourceforge.net/projects/archistyledhtml/files/
  • Extract it to a directory, as you would do with a vanilla Zenta.
  • Start the Zenta executable in the extracted folder.
  • Set a property for the model named report:part with the value all.
  • Go to Edit/Preferences/Styled Export
  • Push the "Install provided styles" button. Choose a folder to install the styles to.
  • For stylesheet location, choose docbookandhtml.style from the folder where you have put the styles to.
  • For output dir, choose a folder where you want the generated documentation to go. You might also want to check the "don't ask for location" checkbox"
  • Choose file/Export/model to Styled HTML....
  • Congratulations, you have a documentation generated! It is called index.html in the folder you have choosen. Take a look at it with your web browser.
  • Now you can continue to explore other possibilities of the module. There are plenty of them

2. Modeling

In the modeling phase, you describe something (the thing which is modeled) based on some ontology (the metamodel), resulting in a model.

The Zenta methodology gives us a metamodel, which is directly useable with Zenta. However often you need a metamodel more detailed than that.

1. Simple uses

For simple uses you don't even need a metamodel.

Creating docbook (and html) documentation

In some cases you just want to create a documentation which contains all, or some parts of the model.

The docbook style - which is the successor of the structured style - does just that for you.

There are actually two style files shipped: docbook.style creates index.docbook and docbookandhtml.style creates index.docbook and index.html.

The documentation will contain the directories as docbook sections, with the diagrams first, then all the contained elements in a table as rows. The element documentation contains the element properties, except the ones known by the style and objectClass.

The style is able to convert a very limited subset of html markup in the documentation to docbook (and then html). Beware that anything looking like html and not in that subset is silently omitted from the output. As too much element documentation means too few modeling, I won't go into more details, see unescape.xslt. If you really need more, please submit a working github pull request with a test case and I might include it.

The style is customizable in two ways:

  • model properties: there are properties which define which parts of the model will be actually generated. With the unmodified style, you should set the report:part property to all for the model or a directory.
  • stylesheet properties: if you open the style file with a text editor, you will see a line containing the name of the xml stylesheet (docbook.xslt) doing the bulk of the work, and some parameters passed to the stylesheet. These are customizable using any text editor. I recommend saving the style file in another name in the same directory before that.

If you would need any more customisation of the style, please file a bug report.

2. Metamodels

Every model have some ontology as a background. In a lot of cases it is implicit, but there are cases when it is better to define it exactly. For example a well defined ontology is the basis for any model driven code/configuration generation.

Another case is when you want to check your or others' work for mistakes and omissions.

These cases need a well-defined set of rules about the subject matter. The ontology is captured in Zenta as a metamodel, and based on it a policy is generated. The policy is then used by the rich export functionality to emit warnings about nonconformances, and to export the model in a way which is easily used by stylesheets to generate the necessary data.

1. Defining a metamodel

The current way to define the metamodel is not considered to be the final one, however most probably I will keep this method as well. In the future metamodels will be defined by template diagrams, where the relations of a template object will define its attributes and the way the attributes obtain their value. Also the extension methods defined by Zenta 2.0 will be fully taken into account. The concepts are already very similar (maybe v2.0 was partly modeled by the Extended Zenta concept, see the history section), but I yet to check that the standard is actually followed.

The core concept of the metamodel is objectClass. The objectClass tells us that we are facing a specialisation of an Zenta object (or any object defined by Zenta, like Folder or model). As an example, we can have an Application Service with the objectClass WebService. In the metamodel level an objectclass have a parent (a built-in objectclass or a descendant thereof), and maybe some attributes. The attributes themselves have various properties, which may have default values.

In the next sections we figure out how to define the objectClass for an Zenta object, and we learn how to define a metamodel.

1. Ways to define the objectClass

define the objectClass by direct property An objectClass can be directly defined for an element by setting an objectClass property with the value as the objectClass name
define the objectClass by indirect property Any property having a : in its name implies that the string before the : is the objectClass of the containing element.

For example by setting report:part to all on a folder, we actually telling that the folder have a report objectClass, and a report specific attribute named part with the value all.
define the objectClass by specialisation this does not work yet.

If object A is of the objectClass Template then all objects which specialisation of A will have the objectClass named after the name of A.
define the objectClass with a Group or Folder By setting the associatedObjectClass property of a Group or a Folder, we declare that all elements in it are belonging to the objectClass named by the value of the property.

2. Creating a policy, checking policy conformance, quick export for xml-aware tools

You need policy.style for that. It uses the "built-in" policy (generated from this model) to generate a policy named "policy.xml" in the target directory, then uses the generated policy on the model, exporting "archirich.xml" an object list in html and docbook format, and a couple of cvs files in the csvdir directory.

Any policy violation is logged as an error. As an example running policy.style in this model should result in exactly one error in both rounds; there is a deliberate omission in the model to show this feature. You can click on the log and it navigates you to the offending element.

The file "archirich.xml" will contain elements for each instances of extended zenta objects. These elements are named after the objectClass's name, and contain one element for each of the attributes: the informations are already assembled (maybe from parts of the model far away from each other) for use by xml-aware tools.

An example of such xml-aware tools is the xslts running in the next step, creating the objectlist docbook and html files contain the list of objects, as well as csv files for each objectClass.

If you need anything more (for example harsher error handling) just drop a bug report.

6. history

Extended Zenta

The Zenta 2 standard have introduced methods to extend the underlying metamodel of Zenta.

The StyledHtml module will go along these lines.

For historical purposes you can find my proposal from Zenta 1 times here:

This is a proposal for an extension to Zenta methodology, adding a metamodel layer in the following way:

- Instances of Zenta objects can have properties. This extension is already implemented in Zenta.

- There is a 4th layer above the well-known ones: the ontology/metamodel layer, describing concepts derived from core zenta concepts, and the properties associated with that concept.

- A new concept of the ontology should be either a simple one, or a composite one. There are the following constraints to derive concepts of an ontology:

- A simple concept should be derived from a core concept of the Zenta ontology.

- A composite concept is a structure built from simple concepts. A composite concept is a "macro": it is defined by and can always be replaced with a structure of simple concepts, and all properties of the composite concept can be bound to a simple concept in the underlying structure.

As you can see Extended Zenta is not another arbitrary methodology, because:

- A model described in the extended zenta methodology can always be described in the core zenta methodology. Moreover a model described in ontology B can be described in ontology A, if B is derived from A.

- Because of the above, all reasoning which holds for the core zenta methodology will hold for any ontology B based on it. Moreover for each ontology A, and B where B is derived from A, all reasoning for A will hold for B.

- If there would be a methodology which incompatible with zenta - in the sense that models using that methodology cannot be meaningfully described in zenta - it would not be describeable in the extended methodology either (but zenta is capable to describe everything, so it is merely a theoretical point:).

- Models using other methodology could be described by first defining the ontology of the other methodology in EA ontology layer, making zenta the mother of all methodologies:)

Reasoning for the necessity of EA:

Zenta right now is a closed ontology. It does have a lot of advantages, and EA would not loose any of them, because any model in EA should have a representation in the core ontology. But a closed ontology is not capable to describe our world in the necessary detail . When you start any design work, the first step is to create a view of the world around you. Let's take a sentence from the standard itself to illustrate it: "Examples of business actors are humans, departments, and business units." So to have the descriptive power, the ontology should be open "downward".

This means that zenta as it is would not be able to support a design beyond a point. The more you focus only on some aspects of the world, the deeper you can go using Zenta, but it comes with the cost that you should neglect other aspects, otherwise they would interfere with your model. And there is invariably a point where another methodology should be used, and the model you built can no longer fully serve the purpose of being a reference point. When you gave your model to a developer and he should rebuild it in UML (or just don't have a frame of thought to develop your model further), it is not just a waste of effort, but makes communication harder because the differing views on the world. For us security people who should care with all the layers and from the conception to the details, this is a nightmare.

With extended zenta the whole development cycle would be traceable in one model, refining not just the model but the ontology with it as development progresses. This makes automated checks for model structure and content reasonable, and even generation of code and configuration is possible. This not just gives the opportunity to help the uninitiated to make quality work, and give the tedious work to machines, but also helps traceability across the different levels of detail. If you ever tried to figure out whether a high-level requirement is fully implemented across a complex system, you will know what I am talking about.
