install the StyledHtml package Just unzip the package in the install directory of Zenta. Further info for troubleshooting, consider it only if something goes wrong: The package contains two jar files, which go to the "plugins" directory. The Zenta directory should already contain a "plugins" directory. You don't have to unzip the jar files. If you see new directories with names like "bin" and "META-INF" then you have managed to unzip the jars...

Configuration This is how you configure the StyledHtml module to be able to create nice reports with it.
Configure output directory Use the modify button and the file dialog to designate the directory you want the output be. If you check the "Don't ask for location" checkbox, then styled export won't ask you for the location when exporting. If you don't check it, the directory you have given here will be the default.
Configure the style used You may want to checkout the provided styles before doing this. Press the modify button on the stylesheet location. Give the stylesheet location in the file dialog. Navigate to the stylesheet drectory, and choose the file named "style.xslt".
Create your own style You can create your own style using XSLT. Create a directory, place a file called "style.xslt" in it. The input of style.xslt is a rich format model, the output is "index.html". The contents of the directory containing style.xslt will be recursively copyed to the output directory, so you can place css, images, etc to the style directory. There are also more complex scenarios, see FIXME to learn about them. You can use one of the provided styles as a starting point.
copy-image inserts a diagram to where it is referenced
from-folder This property holds the id of a folder, and folders can be marked with it. All elements from the referenced folder will be added to the documentation of the referring folder, but diagrams, sketches and folders. Yes, it is the id of the model element which is a bit complicated to figure out (you should look at the model's xml). It does not change if you change the name of the element, and it is not ambigous. If you can come up with a better way, and you actually need it then file a bug report.
report:part This property tells us which part of the model to gerenate. There is a corresponding stylesheet parameter named part. The documentation by default will recursively contain all elements in folders below the marked model/folder (see also the noroot stylesheet parameter).
report:role This parameter is to restrict the visibility of some parts of the model, based on the target audience. There is a corresponding stylesheet parameter named role. The objects and/or folders marked with report:role will be generated only if the stylesheet parameter's value is the same as the property's value. It is not possible to use more than one role neither as a property nor as a stylesheet parameter. If you would need this, please file a bug report. If you also want to handle diagrams properly, you have to utilize the split xml stylesheet. There is a corresponding xml tag to mark parts of the element documentation named only, with the role attribute, used in the obvious way.
noroot If this parameter is set, then the folders below the one marked with the report:part property will be documented.
part See the report:part model property.
role Please see the report:role property.

define the objectClass by direct property An objectClass can be directly defined for an element by setting an objectClass property with the value as the objectClass name
define the objectClass by indirect property Any property having a : in its name implies that the string before the : is the objectClass of the containing element. For example by setting report:part to all on a folder, we actually telling that the folder have a report objectClass, and a report specific attribute named part with the value all.
define the objectClass by specialisation this does not work yet. If object A is of the objectClass Template then all objects which specialisation of A will have the objectClass named after the name of A.
define the objectClass with a Group or Folder By setting the associatedObjectClass property of a Group or a Folder, we declare that all elements in it are belonging to the objectClass named by the value of the property.

order The order in which the defaults are evaluated. The default evaluation goes in the following way: 1. i:=0 2. if there is a default with order i, then try it. else stop. 3. if the default gave back a value, then stop. 4. i:= i+1 5. continue from 2. Beware: if defaults are not numbered sequentially from zero, then some of them might not be tried. The reason of this behaviour that the sequence of getting the attributes from the policy programatically is not guaranteed to be the same as they are described there. The property is represented by the attribute with the same name in the policy file.

Create a report You want to configure the style used and maybe the output dir before you do this. You use the Styled Export menu entry. If you did not check "Don't ask for location", then a file dialog wil appear, where you can define the output directory. If the directory already exists (and you did not check the "Don't ask for location" checkbox), a dialog will appear asking whether to overwrite it.
Edit your model You do it in the usual way in Zenta. See http://archi.cetis.ac.uk/download/latest/Zenta%20User%20Guide.pdf
Preview output Using the browser you can preview the report. Just point to index.html in the output dir.
Preview the export log The export log appearsh in an eventlog window. It is most useful if you have a policy related to the report. You can use it to remediate inconsitencies in your model. Clicking on the links will focus on the element which have the problem. Hint: By clicking analysis on the properties tab, you will see the views where the element participates in, and the relations of it. You can navigate to the view by clicking on it.
Analyse problem You can look at several places to figure out what have happened.
Being happy Problem solved, congratulation.
Look at the application log See application log file for the location of application log file. Look at the end of it for any hint.
Look at the problem report dialog The problem report dialog should give you some hint on what happened
No clue If you don't know what caused the problem, or how to solve this, it is time to report it.
Problem identified If you could identify the cause of the problem, you are happy. Now the next step is to remediate or report the problem.
Problem occured If there is something wrong in the process, a problem report dialog will appear. See Troubleshooting for this case.
Remediate problem If you can remediate the problem (e.g. it is a problem with a stylesheet or policy you have created, or there is a known workaround for a known bug), you do it, and you became very happy. If the fix needs something more (it is an unknown bug, or needs a yet unknown fix or change in the source code), please report it.
application log file In the installation directory of Zenta there is a file called Zenta.ini Look at the definition of osgi.instance.area, looking something like this: -Dosgi.instance.area=@user.home/uk.ac.bolton/Zenta The @user.home here is your home directory. Within the directory defined by osgi.instance.area there is a file called .metadata/.log This is the log file.
look at stdout/stderr If still no solution, look at the stdout/stderr of the application. In unix, you should start it from a terminal window to achieve it, or look at the .xsession-error or similar file. See the configuration of your desktop for further details. In windows? Dunno.
report problem You can report any problem through https://sourceforge.net/p/archistyledhtml/tickets/
